# Chicago Taxi Analytics

A dbt-based analytics project that transforms Chicago Taxi Trips data into reusable analytical models and business-facing insights.

The project answers three core analytical questions and provides additional actionable insights through curated marts and a public Looker Studio dashboard.

---

## 1. Project Overview

This project uses **dbt** to transform Chicago Taxi Trips data into a structured analytics layer following a staged data modeling approach:

```text
Raw Source Data
      │
      ▼
  Staging Layer
      │
      ▼
Intermediate Layer
      │
      ▼
   Mart Layer
      │
      ▼
Looker Studio Dashboard
```

The project is designed to answer:

1. Who are the top 100 tip earners — taxi IDs that earn more tips than others over the latest 3-month period?
2. Who are the top 100 overworkers — taxi IDs that work longer hours, regularly work long shifts, and do not take sufficient breaks over the latest 3-month period?
3. Do US federal public holidays have an impact on taxi activity, demand, revenue, and driver earnings?
4. What additional actionable insights can be derived from the analysis?

# 5. Data Model

## 5.1 Staging Layer

### `stg_taxi_trips`

**Grain:** One row per taxi trip.

The staging model provides a cleaned and standardized representation of the source Chicago Taxi Trips data.

Typical transformations include:

* Trimmed whitespace from key identifier and text fields (`unique_key`, `taxi_id`, `payment_type`, `pickup_location`, `dropoff_location`)
* Standardized `company` into `taxi_company` — uppercased and stripped of non-alphanumeric characters to consolidate inconsistent naming
* Replaced null `fare`, `tips`, `tolls`, and `extras` with 0
* Deduplicated records using `SELECT DISTINCT`

Records are excluded where:

* `unique_key` or `taxi_id` is null or blank
* `trip_start_timestamp` is null
* `trip_seconds` is 30 seconds or less (removes negligible/likely erroneous trips)
* Total charges (`fare` + `tips` + `tolls` + `extras`) are 0 (removes trips with no recorded financial activity)

The staging layer is intentionally kept close to the source data and does not contain significant business logic — filters here address data quality (nulls, duplicates, invalid records), not business rules like session/shift definitions, which are handled downstream in the intermediate layer.

---

## 5.2 Intermediate Layer

### `int_taxi_daily_metrics`

**Grain:** One row per taxi per active trip date.

This model aggregates individual taxi trips into daily taxi-level metrics, including trip volume, distance, duration, revenue, driver earnings, per-trip ratios, day of week, and US federal holiday context. 
See `schema.yml` for the full column list and descriptions.


### Revenue Definition

For this analysis:

```text
Total Revenue = Fare + Tips + Tolls + Extras
```

### Driver Earnings Definition

```text
Driver Earnings = Fare + Tips
```

These definitions are used consistently throughout the analytical models.

---

### `int_taxi_work_sessions`

**Grain:** One row per taxi work session.

A work session is created by grouping consecutive trips from the same taxi when the break between trips does not exceed the session boundary.
See `schema.yml` for the full column list and descriptions.

### Session Boundary Assumption

A break of **more than 4 hours** is considered sufficient to start a new work session.

Therefore:

```text
Break > 3 hours → New session
Break ≤ 3 hours → Same session
```

This threshold is an analytical assumption intended to distinguish between short operational breaks and the end of a working period.

### Long Shift Definition

A session is considered a long shift when:

```text
Session duration > 12 hours
```

```text
is_long_shift = 1
```

otherwise:

```text
is_long_shift = 0
```

### Short Break Definition

A break is considered a short break when:

```text
Break before session < 8 hours
```

```text
is_short_break = 1
```

otherwise:

```text
is_short_break = 0
```

The 8-hour threshold is used as an indicator of insufficient recovery time between working sessions.

* State-specific holidays
* City-specific holidays

# 7. Macros

## `get_rolling_period`

The project includes a reusable macro for dynamically calculating a rolling analysis period.

The macro determines:

```text
period_end_date = Maximum available date
period_start_date = Maximum available date - N months
```

For the 3-month analysis:

```text
period_start_date = max available date - 3 months
period_end_date = max available date
```

This macro is used by the Top Tip Earners and Top Overworkers marts.

This ensures that the analysis automatically updates as new data becomes available rather than relying on hardcoded dates.

# 8. Mart Models

## 8.1 `mart_top_earners_3m`

### Business Question

> Who are the top 100 "tip earners" — taxi IDs that earn more tips than others over the latest 3-month period?

### Analysis Period

The model uses a rolling 3-month period based on the latest available trip date.

### Definition of Tip Earner

Tip earners are ranked using total tips earned during the analysis period.

### Ranking Methodology

Taxi IDs are ranked using `DENSE_RANK()`.

The primary ranking metric is:

```text
Total Tips
```

The model also provides rankings by:

* Total Revenue
* Total Driver Earnings
* Tips per Trip

### Why Total Tips?

The assessment asks for taxi IDs that earn more money through tips.

Therefore, the primary ranking uses **total tips earned**, rather than tips per trip.

However, tips per trip and other efficiency metrics are included to provide context and distinguish between:

* High-volume drivers
* High-tip-per-trip drivers
* High-value drivers

### Dashboard Output

The Looker Studio dashboard displays the top 100 ranked taxi IDs and provides additional comparisons between:

* Total tips
* Total trips
* Active days
* Tips per trip
* Tips as a % of Fare
* Tips per active day

---

# 8.2 `mart_top_overworkers_3m`

### Business Question

> Who are the top 100 "overworkers" — taxi IDs that work longer hours, regularly work long shifts, and do not take sufficient breaks?

### Analysis Period

The model uses the latest rolling 3-month period based on the latest available session timestamp.

### Overworker Indicators

The model evaluates drivers based on:

#### 1. Total Working Hours

Total duration of all work sessions during the analysis period.

#### 2. Long Shift Percentage

The percentage of sessions classified as long shifts.

A long shift is:

```text
Session duration > 12 hours
```

#### 3. Short Break Percentage

The percentage of sessions preceded by a break of less than 8 hours.

A short break is:

```text
Break before session < 8 hours
```

### Overworker Score

The overworker score combines three components:

```text
Overworker Score =
    50% × Working Hours Percentile
  + 30% × Long Shift Percentage
  + 20% × Short Break Percentage
```

The working hours percentile is calculated using `PERCENT_RANK()` across taxi IDs.

This scoring approach gives the greatest weight to total workload while also considering sustained long shifts and insufficient breaks.

### Ranking

Taxi IDs are ranked using `DENSE_RANK()` based on the calculated overworker score.

### Dashboard Output

The dashboard highlights:

* Top overworking taxi IDs
* Total working hours
* Long shift percentage
* Short break percentage
* Average break duration
* Trips per session
* Working hours per session
* Overworker score

The purpose is to identify drivers who may face elevated fatigue or operational risk.

# 8.3 `mart_holiday_activities`

### Business Question

> Do US federal public holidays have an impact on taxi activity?

The model compares taxi activity on US federal holidays against a pre-holiday baseline.

### Holiday Metrics

The model measures:

* Active taxi count
* Total trips
* Total driver earnings
* Total revenue
* Total miles
* Total trip hours
* Driver earnings per trip
* Revenue per trip
* Trips per taxi

### Baseline Definition

For each holiday, a separate baseline is calculated.

The baseline period is:

```text
Holiday date - 28 days
through
Holiday date - 1 day

(inclusve of start and end dates)
```

Other federal holidays are excluded from the baseline.

Therefore, the baseline represents the average daily performance across the 28 preceding **non-holiday days**.

### Why a 4-Week Baseline?

A 4-week baseline was selected to capture typical recent activity while covering approximately four complete weeks of operational patterns.

This helps reduce the influence of:

* Day-of-week effects
* Recent operational changes

### Holiday Impact

Percentage change is calculated as:

```text
(Holiday Metric - Baseline Average)
/
Baseline Average
```

The model calculates percentage changes for:

* Active taxi count
* Total trips
* Total driver earnings
* Total revenue
* Total miles
* Total trip hours
* Driver earnings per trip
* Revenue per trip
* Trips per taxi

### Interpretation

A positive percentage indicates that the holiday metric was higher than the baseline.
A negative percentage indicates that the holiday metric was lower than the baseline.

# 9. Analytical Question 1 — Top 100 Tip Earners

## Question

The top 100 taxi IDs can be identified using:

```text
rank_by_total_tips <= 100
```

This allows the business to distinguish between drivers who generate high total tips because of high trip volume and drivers who generate high tips per individual trip.

The analysis can help identify high-performing drivers and understand the behaviours associated with stronger tip generation.

Potential applications include:

* Driver recognition programs
* Performance benchmarking

* Identifying high-value operating patterns

# 10. Analytical Question 2 — Top 100 Overworkers

## Question

> Who are the top 100 "overworkers", taxi IDs that work more hours than others without taking at least 8 hours break and regularly have a long shift?

## Answer

The `mart_top_overworkers_3m` model identifies taxi IDs with higher workload and potential fatigue risk.

The analysis considers:

* Total working hours
* Long shift percentage
* Short break percentage
* Average break duration
* Trips per session
* Working hours per session

Taxi IDs are scored using the weighted overworker score:

```text
50% Working Hours Percentile
30% Long Shift Percentage
20% Short Break Percentage
```

The top 100 taxi IDs can be identified using:

```text
overworker_rank <= 100
```

## Business Value

This analysis can help identify drivers who may have elevated fatigue or operational risks.

Potential applications include:

* Driver welfare initiatives
* Schedule optimization
* Fatigue management
* Encouraging sufficient recovery periods
* Targeted driver interventions

### Important Assumption

The analysis uses trip timestamps as a proxy for working time.

A work session does not necessarily represent the driver's exact working hours because:

* The driver may be waiting for passengers
* The driver may take breaks between trips
* The taxi may be operating without recorded trips

Therefore, session duration should be interpreted as an estimate of observed working activity rather than a precise measure of actual working time.

# 11. Analytical Question 3 — Holiday Impact

## Question

> Did US public holidays impact taxi trips?

## Answer

The `mart_holiday_activities` model compares each federal holiday against its own 28-day non-holiday baseline.

The analysis examines changes in:

* Total trips
* Active taxi count
* Driver earnings
* Gross revenue

This allows the analysis to distinguish between changes caused by:

1. Increased or decreased demand
2. Changes in the number of active taxis
3. Changes in revenue/earnings generated 

## Business Interpretation

The majority of US federal holidays show a decrease in taxi trips, active taxis, revenue, and driver earnings compared to the 4-week pre-holiday baseline — confirming that public holidays do have a measurable impact on taxi demand, and the effect is predominantly negative.

One notable exception — Inauguration Day: Trip counts and active taxi counts both increase slightly (~+2% and +5%), but revenue and driver earnings decrease (~-8%). More taxis were on the road doing more trips, but earning less — suggesting shorter average trips or lower fares per trip on that day (e.g., concentrated short-distance activity around a specific event/area), rather than a broad demand increase.

## Business Value

The analysis can support:

* Holiday driver incentive planning
* Driver supply forecasting
* Demand planning
* Targeted promotions
* Identifying holidays with unusually high or low demand

A key actionable opportunity is that active taxi supply tracks demand closely across most holidays — meaning driver incentives are best targeted not at closing a supply gap, but at reducing planned supply ahead of low-demand holidays (e.g. Labor Day, Independence Day) to avoid oversupply, while maintaining normal staffing for holidays that show flat or positive demand (e.g. Juneteenth, Columbus Day). Inauguration Day stands out as an exception — trip and active taxi counts rose while revenue and driver earnings fell — warranting further investigation into whether this reflects lower-fare, shorter trips concentrated around a specific event or area, which could inform targeted repositioning rather than a supply change.

# 12. Bonus Analytical Insights

## Insight 1 — Overworking does not correlate with higher revenue

### Insights
Cross-referencing mart_top_overworkers_3m and mart_top_earners_3m shows almost no overlap between the two lists — of the top 10 overworkers, only 1 taxi (overworker rank #10) also appears in the top 10 by revenue (rank #3). The remaining 99 top overworkers don't place highly on the revenue leaderboard at all. 

### Business Value

This decouples "hours worked" from "revenue generated" — a distinction that matters because it changes what action the business should take. If overworking did correlate with earnings, you might justify it as a rational (if risky) trade-off drivers are making. Since it largely doesn't, the long hours and poor rest patterns among this group represent fatigue and compliance risk without a corresponding return, meaning any intervention (rest enforcement, shift caps) has close to zero revenue trade-off cost. The business could reduces safety/liability exposure without giving up meaningful earnings from these drivers.

## Insight 1 — Overworking does not correlate with higher revenue

### Insights
Cross-referencing `mart_top_overworkers_3m` and `mart_top_earners_3m` shows almost no overlap between the two lists — of the top 100 overworkers, only 1 taxi (overworker rank #10) also appears in the top 10 by revenue (rank #3). The remaining 99 top overworkers don't place highly on the revenue leaderboard at all.

### Business Value

This decouples "hours worked" from "revenue generated" — a distinction that matters because it changes what action the business should take. If overworking did correlate with earnings, you might justify it as a rational (if risky) trade-off drivers are making. Since it largely doesn't, the long hours and poor rest patterns among this group represent fatigue and compliance risk without a corresponding return, meaning any intervention (rest enforcement, shift caps) has close to zero revenue trade-off cost — the business can reduce safety/liability exposure without giving up meaningful earnings from these drivers.

### Recommended Action

Prioritize the top 100 overworkers list (rather than the fleet broadly) for a targeted rest-compliance intervention — e.g. a mandatory minimum break rule, or an automated alert when a driver's session pattern crosses the long-shift/short-break thresholds used in this analysis. Because this group's revenue contribution is not meaningfully above average, enforcing rest limits here is unlikely to hurt total fleet revenue, making it a low-risk starting point to pilot before considering a fleet-wide policy.

## Insight 2 — Top earners aren't always the most efficient earners

### Insights

Ranking taxis purely by total_driver_earnings conflates two very different behaviors — driving a high volume of trips, versus generating strong earnings on each individual trip. Comparing `total_driver_earnings` against `driver_earnings_per_trip` for the same top-100 taxis shows these two rankings diverge, some taxis land in the top earners list mainly by racking up trip count (total_trips), while their `driver_earnings_per_trip` is only average. Conversely, some taxis with comparatively fewer total trips show a notably higher `driver_earnings_per_trip`, but don't rank as high on total earnings simply because they drove less overall. 

### Business Value

This distinction matters because "high earner" and "efficient earner" call for different business actions. A taxi generating high revenue purely through volume is essentially trading time for money. It is not sustainable because it is not necessarily improvable without working more hours (which, per the overworker analysis, has real fatigue/compliance cost). A taxi with high `driver_earnings_per_trip` is doing something more valuable per unit of time or trip, perhaps better routes, better positioning during high-fare periods, stronger customer service driving tips, or picking up longer/higher-fare trips. Identifying this second group is where the real efficiency insight is as their behavior is more scalable, also worth understanding and replicating across the fleet (e.g. through driver coaching, route/zone recommendations, or shift-timing guidance), rather than the volume-driven group, whose earnings would mostly just scale by working more hours.

### Action
Isolate the "efficient earner" segment specifically — taxis with high `driver_earnings_per_trip` but moderate `total_trips` — and treat them as a source of best practices rather than just a ranking curiosity. Cross-reference their trip data (e.g. pickup/dropoff zones, time-of-day patterns) to identify what they're doing differently, then package those findings into concrete driver guidance (e.g. recommended zones or shift windows). Pilot this guidance with a sample of volume-driven top earners and track whether their `driver_earnings_per_trip` improves in the following period — this directly tests whether efficiency behavior can be taught, rather than assuming it and moving on.

# 18. Limitations and Assumptions

### Taxi ID as Driver Proxy

The analysis assumes that a `taxi_id` can be used as a proxy for a taxi/driver entity.

A taxi ID may not always represent a single individual driver.

### Working Hours

Working hours are estimated based on observed trip timestamps and session boundaries.

They should not be interpreted as exact clocked working hours.

### Session Boundary

A break of more than 4 hours starts a new work session.

This is an analytical assumption and may not perfectly represent actual driver work patterns.

### Long Shift

A session longer than 12 hours is classified as a long shift.

### Short Break

A break shorter than 8 hours before a new session is classified as a short break.

### Holiday Scope

Only US federal holidays from 2013 to 2023 are included.

### Holiday Baseline

The baseline uses the 28 preceding non-holiday days for each holiday.

### Rolling 3-Month Period

The Top Tip Earners and Top Overworkers analyses use the latest available 3-month period dynamically determined from the data.
